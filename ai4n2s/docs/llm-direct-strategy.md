# LLM-Direct 策略设计文档

> 本文档详细说明 `llm-direct` 剧本生成策略的设计理念、工作流、提示词工程和容错机制。

## 概述

`llm-direct` 是 AI4N2S 系统中三种剧本生成策略之一。它采用**两阶段生成 + 智能重试**的设计，直接将结构化小说数据（`NormalizedNovel`）转化为标准剧本格式（`ScriptYAML`），不依赖向量数据库或 RAG 检索。

### 策略定位

| 策略 | 依赖 | 适用场景 | 质量 | 延迟 |
|------|------|---------|------|------|
| `default` | 无 | 快速初始化、空白模板 | 低 | 毫秒 |
| `llm-direct` | LLM | 中等规模小说，无需检索增强 | 中高 | 分钟 |
| `ai-rag` | LLM + RAG | 长篇小说，需要上下文检索 | 高 | 分钟 |

## 两阶段设计

### 阶段 1：草稿生成（Draft）

**目标**：将小说叙事文本转化为初版剧本格式。

**参数配置**：
- `temperature: 0.7` — 较高温度保证创意多样性和文本流畅度
- `maxTokens: 4000` — 足够生成 3-10 个内容项

**Prompt 结构**：

```
[SYSTEM] 你是一位资深影视编剧...（格式规范 + 写作要求 + 输出约束）
[USER]
  【章节标题】{title}
  【章节内容】{content.slice(0, 6000)}
  【全书摘要】{plot_summary.slice(0, 500)}
  【角色档案】
  1. 罗辑【主角】— 社会学博士 — 性格: 玩世不恭
  2. 大史【配角】— 资深刑警 — 性格: 粗中有细
  ...
```

**设计要点**：

1. **角色档案内嵌**：将全部角色信息（最多 15 个）直接编入 prompt，确保 LLM 在生成对话时了解角色性格和定位。避免 RAG 检索的不确定性。

2. **内容截断到 6000 字**：`slice(0, 6000)` 保证 prompt 在 LLM 上下文窗口内。对于绝大多数字章节，6000 字覆盖了核心内容。

3. **全书摘要辅助**：2-3 句话的情节概览帮助 LLM 理解当前章节在全书中的位置。

4. **严格的 JSON 输出约束**：系统提示词中反复强调"只返回 JSON 数组，不包含任何解释"，减少后处理负担。

### 阶段 2：审查润色（Review）

**目标**：检查草稿质量并自动修正问题。

**参数配置**：
- `temperature: 0.3` — 较低温度保证输出稳定和一致性
- `maxTokens: 4000` — 与草稿阶段相同

**审查维度**：

| 维度 | 检查内容 | 处理方式 |
|------|---------|---------|
| 角色声音一致性 | 同一角色台词风格是否统一 | 微调措辞 |
| 对话自然度 | 台词是否生硬或拗口 | 优化表达 |
| 格式规范性 | action/character/transition/shot 使用正确性 | 修正类型 |
| 情节连贯性 | 与前一个场景的过渡 | 补充转场 |
| 文学质量 | 动作描述生动性 | 润色 |

**优化原则**（Prompt 中明确规定）：
- 保留草稿优点，只修改确实有问题的地方
- 不要大幅改写
- 对话微调使其符合角色性格
- 如果格式类型用错，修正它

**为什么先草稿后审查，而非一次生成？**

单次 LLM 调用同时处理"创意生成"和"质量审查"会导致认知负载过高：temperature 高则创意好但错误多，temperature 低则准确但缺乏灵感。两阶段分离让每个阶段专注于单一目标。

### 跳过审查的条件

```typescript
// 草稿只有 1 个 action 项时跳过审查
if (draftContent.length <= 1 && draftContent[0]?.type === 'action') {
    return draftContent;
}
```

这涵盖了两种情况：
1. LLM 调用失败返回的占位内容（无需审查）
2. 章节内容为空导致的单 action 输出（无需审查）

此外，审查前会对草稿进行裁剪（最多 15 项，每项截断到 300 字），防止 prompt 超出上下文窗口。

## JSON 解析容错

LLM 输出不可靠，必须进行多级容错：

```
Level 1: 直接 JSON.parse(content)
            │ 失败
            ▼
Level 2: 提取 ```json ... ``` 或 ``` ... ``` 代码块
            │ 失败
            ▼
Level 3: 正则提取 [{...}] 或 {...} 片段
            │ 失败
            ▼
Level 4: LLM 重试（附加严格指令："只返回 JSON 数组！"）
            │ 失败
            ▼
Level 5: 原始文本作为 action 包裹（兜底）
```

### 重试策略

```
第 1 次 LLM 调用: 标准 prompt，temperature=0.7
    │ JSON 解析失败
    ▼
第 2 次 LLM 调用: 附加警告 + 严格指令，temperature=0.3
    │ JSON 解析失败
    ▼
兜底: [{ type: 'action', text: llmRawContent.slice(0, 2000) }]
```

两次调用的参数差异：
- 温度从 0.7 → 0.3（降低随机性，提高格式遵循度）
- Prompt 增加警告前缀（强调格式要求）
- 两次调用之间不做内容修改，仅调整指令

## 角色一致性保障

### 角色档案构建

```typescript
function buildCharacterProfile(characters: Character[]): string {
  return characters.slice(0, 15).map((c, i) => {
    const parts = [`${i + 1}. ${c.name}`];
    if (c.role) parts.push(`【${c.role}】`);
    if (c.description) parts.push(c.description);
    if (c.personality) parts.push(`性格: ${c.personality}`);
    if (c.aliases?.length) parts.push(`别名: ${c.aliases.join('、')}`);
    return parts.join(' — ');
  }).join('\n');
}
```

输出示例：
```
1. 罗辑【主角】— 社会学博士 — 性格: 玩世不恭、善于自嘲 — 别名: 罗教授
2. 大史【配角】— 资深刑警 — 性格: 粗中有细、重情义
```

这份档案在草稿和审查两个阶段都作为 prompt 的一部分传入，确保跨场景的角色一致性。

### 情节连贯性

审查阶段传入前一个场景的摘要：
```
【前情回顾】前一个场景: 第一章 初见: 张三: 你好...; 转场: CUT TO:
```

审查 prompt 中明确要求检查"情节连贯性：场景内部和场景之间的逻辑是否通顺"。

## 错误处理与降级策略

| 故障点 | 降级方式 | 影响 |
|--------|---------|------|
| LLM API 不可达 | 返回 `[{ action, text: "[LLM 调用失败]..." }]` | 单章占位 |
| LLM 返回非 JSON | 4 级容错 → 最终包裹为 action | 单章劣化 |
| 审查阶段失败 | 保留草稿内容 | 无影响 |
| 单章处理异常 | catch → push 占位场景 | 仅该章受损 |
| 全书无章节 | 生成一个空场景 | 空模板 |

**核心原则**：宁可输出不完美的剧本，也不能让管线崩溃。用户始终可以看到生成结果，即使部分章节需要手动修正。

## 与 ai-rag 策略的对比

| 维度 | llm-direct | ai-rag |
|------|-----------|--------|
| 上下文获取 | 内容截断到 6000 字 | RAG 检索相关片段 |
| 角色信息 | 全部角色内嵌 prompt | RAG 检索 + 角色索引 |
| 长文本处理 | 截断 | 分块检索 |
| LLM 调用次数 | 2 次/章 | 2 次/章 |
| 额外基础设施 | 无 | 向量数据库 |
| 适用章节长度 | ≤ 6000 字效果最佳 | 任意长度 |
| 质量天花板 | 中高 | 高（更多上下文） |

## 性能考量

| 因素 | 数值 |
|------|------|
| 每章 LLM 调用 | 2 次（草稿 + 审查） |
| 每次调用输入 token | ~2000-4000 |
| 每次调用输出 token | ~500-2000 |
| 每章耗时（典型） | 5-10 秒 |
| 11 章小说总耗时 | 约 1-3 分钟 |
| 成本（DeepSeek） | 约 ¥0.05-0.2/章 |

## 未来优化方向

1. **并行化**：独立章节的草稿生成可以并行执行（当前串行是为了情节连贯性）
2. **动态 temperature**：根据章节类型（动作/对话/描写）自动调整
3. **角色一致性外部校验**：用独立 LLM 调用对比不同章节中同一角色的对话风格
4. **增量生成**：只重新生成修改过的章节，而非整本小说
